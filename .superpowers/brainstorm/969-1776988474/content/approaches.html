Abordagens de implementação

Três formas de estruturar a lógica de correção de flags. A lógica de negócio é a mesma — a diferença está em onde a complexidade fica.

A

RPC no Postgres (recomendada)

Uma função correct_exam_flags no banco cuida de tudo atomicamente: atualiza attendances, atualiza queue_items.com_laudo, fecha pipeline_items desmarcados, cria novos pipeline_items para flags adicionadas. A API e o frontend ficam simples — só chamam a RPC e recebem o resultado.

Prós

  • Atômico — não tem estado parcial
  • Consistente com o padrão já usado no projeto
  • Auditoria centralizada no banco

Contras

  • Requer escrever migration SQL
B

Lógica na API Route (TypeScript)

A route PATCH /api/clinic/pipeline-items/[id]/correct-flags faz as operações em sequência: atualiza attendance, atualiza queue_items, fecha/cria pipeline_items. Tudo em TypeScript, sem nova RPC.

Prós

  • Mais fácil de depurar
  • Sem migration SQL

Contras

  • Não atômico — uma falha no meio deixa dados inconsistentes
  • Mais verboso
C

Estender a RPC existente admin-action

Adicionar um novo action: "correct_flags" à route /api/clinic/pipeline-items/[id]/admin-action já existente, reutilizando o padrão de autenticação e auditoria.

Prós

  • Reutiliza infraestrutura existente

Contras

  • Mistura responsabilidades diferentes num endpoint já cheio
  • Ainda precisaria de RPC ou lógica sequencial no banco
